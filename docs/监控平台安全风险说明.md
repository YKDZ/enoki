# Enoki 监控平台安全风险说明

日期：2026-06-24

本文记录本次参考 `docs/软件审计.md` 对 Enoki 监控平台进行安全审计后，不能通过小范围代码补丁立即彻底消除的设计级或运营级风险。可直接修复的工程缺陷已进入代码修复和回归测试流程；本文聚焦剩余风险、影响边界、补偿控制和后续改进方向。

## 审计范围

- Hub 后端：Owner 认证、管理 API、Probe 注册/上报/配置、WebSocket、审计日志、Probe 操作 token。
- Web UI：主要按管理 API 调用面和浏览器会话边界评估。
- Probe：注册、请求签名、bootstrap 配置、升级/卸载、Probe Local Privilege Boundary、Privileged Collector Helper、本地文件权限。
- 安装与发布：安装脚本、Docker 入口、Probe 资产分发、发布工作流和供应链完整性。

## 风险总览

| 编号  | 风险                                             | 等级 | 当前状态                       |
| ----- | ------------------------------------------------ | ---- | ------------------------------ |
| R-001 | Hub 是 Probe 升级/卸载的高信任控制面             | 高   | 设计接受，需强化审计和操作控制 |
| R-002 | 单 Owner 权限模型缺少角色拆分与二次确认          | 中   | 设计接受，适合小型自托管       |
| R-003 | Probe 本地服务用户被攻陷后的横向影响             | 中   | 已拆分提权入口，仍需运营恢复流程 |
| R-004 | 发布链路缺少完整 SLSA/SBOM/镜像签名闭环          | 中   | 需发布工程改进                 |
| R-005 | TLS、反向代理和可信网络依赖部署者正确配置        | 中   | 需文档、启动门禁和部署基线     |
| R-006 | 资产签名密钥轮换和撤销策略不足                   | 中   | 需设计 key rotation runbook    |
| R-007 | Probe Operation token 默认运行期密钥影响重启语义 | 低   | 需生产部署说明或强制稳定密钥   |
| R-008 | 注册和登录滥用检测仍偏基础                       | 低   | 需监控告警和边缘限流           |

## R-001 Hub 是 Probe 升级/卸载的高信任控制面

Enoki 的架构允许 Owner 通过 Hub 对单个 Host 发起 Probe Upgrade 或 Probe Uninstall。Probe Operation 是有限枚举，不支持任意 shell 命令；升级资产也有 manifest 签名、归档校验和、防降级与目标版本校验。这些控制能显著降低远程代码执行面，但无法改变根信任事实：一旦 Hub Owner 会话、Hub 数据目录、Probe 操作 token 签名密钥、资产签名私钥或发布镜像被攻陷，攻击者可能借合法操作面影响已注册 Host。

当前补偿控制：

- Probe Operation 类型有限，不提供任意命令下发。
- Probe 请求使用非对称身份签名，Operation token 绑定 Probe、Operation 和目标版本。
- Probe 升级前校验资产签名、签名公钥指纹、归档哈希和版本防降级规则。
- README 已明确 Hub 权限、Hub 数据目录、资产签名私钥和镜像发布权限属于高信任边界。

建议改进：

- 为升级/卸载增加更明确的敏感操作确认、延迟窗口或本地策略开关。
- 对 Probe Operation 建立更完整的审计日志，包括发起来源、目标 Host、目标版本、token 验证、Probe 侧状态和最终 Host Profile 确认。
- 提供紧急冻结开关，允许部署者临时禁止 Hub 下发新的 Probe Operation。
- 为团队场景设计审批或四眼确认机制。

## R-002 单 Owner 权限模型缺少角色拆分与二次确认

当前 Hub 是单 Owner 模型，认证通过后可以执行所有管理操作，包括创建 Enrollment Token、修改 Probe Configuration、编辑 Host Metadata、触发 Probe Upgrade、触发 Probe Uninstall 和删除 Host。该模型符合 Enoki 当前“小型自托管监控平台”的定位，但不适合多人运维、外包运维或强合规环境。

当前补偿控制：

- 生产和 Docker 环境默认要求设置 `OWNER_PASSWORD`。
- Owner 会话 Cookie 使用 `HttpOnly`、`SameSite=Lax`，HTTPS/反代场景可设置 `Secure`。
- WebSocket live updates 绑定 Owner session，登出时关闭对应连接。
- 关键管理动作已有基础 audit log。

建议改进：

- 引入最小角色集合，例如只读 Guest、Host 管理、Probe Operation 管理。
- 对升级、卸载、删除 Host 等破坏性操作增加二次确认或短期 re-auth。
- 支持会话列表、主动撤销、较短敏感操作授权窗口和设备绑定。
- 将“可信网络内使用”升级为部署基线，而不是仅作为 README 建议。

## R-003 Probe 本地服务用户被攻陷后的横向影响

Probe 常态下不以 root 运行，特权动作通过受限 sudoers/systemd Probe Local Privilege Boundary 完成。这个边界降低了默认运行风险，但如果本地 Probe 服务用户被攻陷，攻击者仍可能读取 Probe 私钥、影响上报内容、修改可写状态、等待 Hub 下发合法 Operation，或尝试影响本地升级状态文件。

当前补偿控制：

- bootstrap 配置运行时读取有权限检查，拒绝 symlink、非 regular file、group/other 可访问文件。
- 安装器会把 canonical Hub URL 写入 root-owned install metadata；root 内部升级/卸载入口在 token validation、资产下载和状态上报前校验 bootstrap Hub URL 与 metadata 一致，并使用 metadata 中的 Hub URL 作为可信目标。
- 特权 collector helper 入口是编译期枚举，Owner 不能通过 Hub 注入任意命令。
- sudoers 入口限制为内置操作，不把 Operation token 放在 argv。
- Probe Operation 和特权 collector helper 使用固定的内置子命令和精确 sudoers 命令行；Operation sudoers 与 collector-helper sudoers 已拆分为独立文件，安装和升级会移除历史 `enoki-probe-upgrader` 混合 sudoers 文件。
- 当前官方特权 collector helper 只有 `disk-health.smartctl`，Probe 内部 helper CLI 只接受已编译进 binary 的 helper id；安装器和升级器都通过已安装 Probe binary 的 `internal-render-collector-helper-sudoers` 入口渲染 helper sudoers，使策略来自目标 Probe 资产的编译期声明。
- collector-helper sudoers 只在安装/升级时按本机轻量 exposure predicate 渲染。当前官方 helper 只检查固定 `smartctl` 路径；没有 helper 暴露时不会保留 collector-helper sudoers 文件。
- Hub、Owner、Probe Configuration、Host Profile、Collector Capability 和 Metrics payload 不能在运行时新增、删除或改写 helper exposure。
- 特权 collector helper 使用 systemd-run，已有运行时长和网络隔离等限制；这不是完整 sandbox，当前没有设备级或路径级隔离。

建议改进：

- 将 Probe 私钥、普通上报状态、Operation 执行状态进一步拆分到不同本地权限域。
- 继续评估 root-owned bootstrap 配置加单独可写状态文件的模型；当前 root 操作目标已由 root-owned install metadata 固定，但 service user 仍可影响自身身份材料和普通上报内容。
- 后续评估每类 privileged collector helper 的 device/path profile，而不是把当前 timeout 和 network profile 误认为完整 sandbox。
- 明确运维流程：如果主机安装后新增或移除 `smartctl` 等可选 helper 前置条件，需要重新安装或升级 Probe 才会重新评估 collector-helper sudoers。
- 为 service user compromise 编写恢复流程：轮换 Probe identity、清理 pending operation、重新 enrollment。

## R-004 发布链路缺少完整 SLSA/SBOM/镜像签名闭环

Probe 资产有 manifest 签名和校验和，Hub 镜像可从 GHCR 拉取。但完整供应链仍依赖 GitHub Actions、GHCR、GitHub Release、签名私钥变量和 tag 管理的组合信任。当前没有完整的镜像 cosign 签名、SLSA provenance、SBOM 发布、不可变 release 策略和密钥轮换演练。

当前补偿控制：

- Probe 资产安装和升级时校验签名公钥指纹、manifest 签名和资产哈希。
- 仓库锁定 pnpm/Cargo 依赖版本。
- README 指出不信任官方发布链时可以 fork 并配置自己的发布密钥。

建议改进：

- 为 Hub 镜像和 Probe 资产发布 cosign 签名与 provenance。
- 每个 release 产出 SBOM，并在发布说明中公开制品哈希。
- 避免覆盖已发布资产，生产部署推荐使用版本 tag 或 digest pinning。
- 建立资产签名密钥轮换、吊销和紧急回滚 runbook。

## R-005 TLS、反向代理和可信网络依赖部署者正确配置

Enoki 明确建议管理界面仅在可信网络中使用，Probe API 公网部署时应使用 HTTPS 或可信隧道。Hub 自身不终止 TLS，Cookie `Secure` 和可信代理头也依赖环境变量正确配置。这是自托管轻量架构的合理取舍，但会把一部分安全结果转移给部署者。

当前补偿控制：

- Docker/生产默认要求 `OWNER_PASSWORD`。
- `ENOKI_PUBLIC_HTTPS` 和 `ENOKI_TRUST_PROXY_HEADERS` 可让反代后 Cookie 使用 Secure。
- Probe 请求签名保护请求体、重放窗口和 canonical origin；启用 trusted forwarded headers 时，Hub 会用可信 `X-Forwarded-Proto` / `X-Forwarded-Host` 计算反代后的公网 origin，默认不信任这些 header。
- 安装脚本要求 HTTPS Hub，localhost 开发例外已收窄到精确 host：`localhost`、`127.0.0.1`、`[::1]`。

建议改进：

- 发布生产部署基线：HTTPS、HSTS、反向代理 header、管理端访问控制、Probe API 暴露策略。
- 启动时检测明显危险组合并失败关闭，例如公网监听、无 TLS 提示、无密码模式。
- 在安装命令生成时更明确区分管理 URL 和 Probe API URL。
- 为反代配置提供 Nginx/Caddy 示例和安全 header 模板；启用可信 forwarded headers 前，部署者仍必须确保这些 header 只能由受信反代写入，不能由公网客户端直达伪造。

## R-006 资产签名密钥轮换和撤销策略不足

Probe 安装和升级使用固定公钥指纹 pinning。该模型能防止资产被替换，但会让签名密钥轮换和泄露响应变成高风险运维流程。如果旧 key 泄露，部署者需要清楚如何阻断、替换和恢复信任链。

当前补偿控制：

- 安装脚本和升级器校验受信公钥指纹。
- Manifest 和归档都有完整性校验。
- 防降级规则降低旧版本重放风险。

建议改进：

- 支持双签或多受信 key 的过渡窗口。
- 发布 key rotation 指南，包含旧 key 吊销、Hub 资产更新、Probe 升级顺序和失败回滚。
- 在 Hub UI 或健康检查中展示当前资产签名 key 指纹和目标 Probe version。

## R-007 Probe Operation token 默认运行期密钥影响重启语义

如果未配置 `ENOKI_PROBE_OPERATION_TOKEN_SIGNING_SECRET`，Hub 会在启动时生成随机 Operation token 签名密钥。该默认值减少了静态密钥暴露，但 Hub 重启会使已经下发、尚未完成验证的 Operation token 失效。对单实例轻量部署可以接受，对多实例或升级窗口较长的生产部署需要明确语义。

当前补偿控制：

- README 已说明多实例或跨重启保留未完成操作时应设置稳定随机值。
- Operation token TTL 短，并绑定 Probe、Operation 和目标版本。

建议改进：

- 生产模式下对未配置稳定 secret 给出更强启动警告或门禁。
- 重启后将 pending/accepted/running operation 标记为需要重新下发或等待 Probe 重新获取 token。
- 在 UI 中展示 Operation 因 Hub 重启/密钥变化导致失败的明确原因。

## R-008 注册和登录滥用检测仍偏基础

Enrollment Token 强随机、短 TTL、一次性消费，Owner 登录也有固定失败延迟和审计记录。但公网部署时仍需要更完整的异常检测，例如注册失败突增、登录失败突增、同一来源大量无效 Probe 签名请求等。

当前补偿控制：

- Enrollment Token 只存 SHA-256 hash，15 分钟过期，成功注册即消费。
- Probe 已注册后的接口要求请求签名、nonce 和时间窗。
- 登录成功/失败有 audit log。

建议改进：

- 对登录、注册、Probe 签名失败增加按来源的速率限制和告警。
- 在 audit log 中统一记录 remoteAddress，并提供最近安全事件视图。
- 推荐在公网入口前部署反向代理或边缘限流。

## 复评条件

以下任一条件发生时，应重新进行安全评估：

- 引入多用户、Guest、RBAC、OAuth 或外部身份提供方。
- 引入任何 runtime plugin、脚本 collector、URL collector、命令型 collector 或 Hub 下发逻辑。
- 改变 Probe Operation 类型、升级器权限、sudoers/systemd 策略或安装路径模型。
- Hub 管理面计划直接公网暴露。
- 发布链路、签名密钥、镜像仓库或 GitHub Actions 权限模型发生变化。
- 发生凭据泄露、Probe 私钥泄露、签名 key 泄露或供应链安全事件。
